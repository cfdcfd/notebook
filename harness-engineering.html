一、为什么做

1.1 业务价值：「能跑长任务」才是真正差异化

短任务没有壁垒

单次问答、补全——任何商业 API 都能做。短任务无法体现自有集群和 Harness 的价值。

长任务才有飞轮

用户能稳定让 AI 自主运行 数小时、持续产出可合并的代码/文档 → 采用率上升 → 集群价值被证明 → 获得更多投入。

1.2 集群定位：算力是战略资产

维度自有 122B 集群商业 API（Claude/GPT）
推理成本低（MoE 仅 10B 激活）高
可控性完全可控：思考预算、并行度、优先级黑盒
数据安全不出域需信任第三方
模型能力上限弱于同代闭源前沿更强
空余算力大量——可投向评估与验证不存在此概念
核心命题：把大量空余算力 可重复地转化为可靠性与产出质量，而不是空转。Harness Engineering 的使命就是完成这个转化。

二、长任务到底死在哪里

失败往往不来自「某一次 JSON 少了括号」，而来自结构性问题。

失败模式 1：一次做太多

Agent 试图一次完成整个任务 → 上下文耗尽 → 留下半成品 → 下一 session 无法接续。

失败模式 2：过早宣布完成

看到已有进展就认为「差不多了」→ 未对照验收标准 → 实际功能残缺。

失败模式 3：长链误差累积

单步小错在多步后累积放大。长链路中未被捕获的错误有雪球效应——因此在链路中间插入检测与修正节点（评估器、测试、lint）的价值极大。

失败模式 4：自我评估偏松

模型倾向于肯定自己的产出，漏掉明显问题。Anthropic 原话："confidently praising the work—even when quality is obviously mediocre."

因此，Harness Engineering 的本质是：为模型提供 记忆、计划、工具验证与反馈闭环，而不仅是更好的单条提示词。

三、核心洞见（外部研究 + 内部推演）

洞见 1：Harness 对表现影响极大

Anthropic 展示：同一模型在不同 Agent 系统/工具链下表现差异可达 数倍。

注意归因：这里的「Harness」涵盖整个 Agent 系统（工具链、多 Agent 编排、提示设计、评估循环），不仅是 Proxy 层格式修复。差距主要来自 Agent 架构与工具集成。

→ 投入 Agent 侧 Harness 的回报可能大于单纯换一个更强的模型。

洞见 2：独立评估器是最大杠杆

把「写代码」和「判代码」分开，比让生成器自我批评有效得多。

但评估器的核心不是「LLM 读代码给意见」——Anthropic 的 Evaluator 通过 Playwright 打开实际页面、点击交互、查 API/数据库。评估质量来自 「工具验证 + LLM 判断」的组合。

洞见 3：结构化产物是跨 session 桥梁

feature_list（JSON）、progress 文件、git history——这些机器可读的产物让新 session 能精确接续，比自然语言聊天记录可靠得多。

洞见 4：Harness 应随模型变强而简化

每个 Harness 组件都编码了「模型做不好什么」的假设。模型升级后要重新验证——Build to Delete。

2026 年 6 月升级 Qwen3.6 / MiniMax2.7 时，当前很多补丁可能应被删除。

4.2 算力投向：评估优先于冗余生成

算力用途优先级理由
Evaluator 评估（工具产出 + LLM 判断）最高独立评估是最大杠杆；评估请求短、GPU 时间少、杠杆大
Planner 任务分解高好的计划决定后续所有工作方向
角色化思考预算中复杂步骤给更多 thinking tokens
空闲时段回归评估中持续校准质量基线
BoN 并行采样低仅在评估器反复不通过时的补充手段
核心转变：从「生成 N 个选最好」→「生成 1 个 → 工具验证 → 独立 LLM 评估 → 基于反馈改进 → 再验证」。
有记忆的评估-反馈循环比无记忆的 Best-of-N 更高效：每轮反馈都缩小搜索空间。

5.1 Agent 框架层

Planner

高层需求 → 结构化 feature 列表（JSON）。专注产品与高层技术设计，不过早细化实现。

对 Proxy：role=planner，高思考预算

Generator

一次一个 feature → 代码 + commit + progress。增量交付，每块留痕。

对 Proxy：role=generator，按 intent 分配预算

Evaluator

先跑工具（测试/lint/浏览器）→ 把工具产出 + 代码交给 LLM 判断 → 输出可定位的缺陷列表与修改建议。

对 Proxy：role=evaluator，触发多视角扇出 + 严格度校准

关键约束：Evaluator 的价值 = 工具验证 + LLM 判断。纯 LLM 静态审阅不是评估，是另一轮猜测。Agent 层必须先落地工具链（测试、lint、浏览器自动化），Proxy 层的评估增强才有意义。

5.2 Proxy 层能力

能力优先级做什么不做什么
角色感知思考预算 P0 按 role + intent 设置 max_thinking_tokens；Evaluator/Planner 高，Generator 按需 不硬编码数值——必须在 Golden Set 上实测调参
评估器增强 P0 多视角扇出、严格度校准（few-shot + 系统提示）、优先级调度 不替代工具验证——LLM 评估建立在工具产出之上
可观测 P1 统一 trace（role/task/feature id）、按角色指标 不做业务层仪表盘（独立服务负责）
模型路由 P1 Qwen ↔ MiniMax 切换、故障降级 不做智能实时路由——起步用规则，后续靠数据
BoN / 约束解码 可选 仅在高价值节点或评估失败时启用 不做全局默认 BoN；不对 122B 全面约束解码

6.2 评估维度

维度LLM 角色输入应包含注意
功能完整性比对验收清单与测试结果测试报告、验收 checklist无测试报告时 LLM 只能静态推断，价值大打折扣
边缘健壮性识别未覆盖的边界条件测试覆盖率、错误处理代码LLM 不能实际运行边缘用例——只能指出缺失防御
代码质量审查结构、命名、重复lint 输出、diff纯静态分析，LLM 相对擅长
安全性（可选）识别常见漏洞模式代码 diff对已知模式有效，对新型攻击无效
规格一致性（可选）对比 spec 与实现原始 spec长任务中模型容易「创造性偏离」

6.3 严格度校准：Few-shot 示例 + 系统提示

层 1：Few-shot 示例（主力）

在评估请求中附带 2-3 个标注示例，展示「这段代码有什么问题、应该给什么判定」。

Anthropic 实际使用的就是这种方法。仅靠"你要严格"等抽象指令效果有限。

层 2：系统提示（辅助）

补充规则：报告每个 issue、测试通过≠代码正确、默认 FAIL 除非有正面证据。

7.2 提示词推理增强：10 个可做的点（P0 — 投入最低见效最快）

无需修改模型权重，无需改推理引擎。Proxy 在转发请求时注入提示片段即可。详见 prompt-reasoning-enhancement.md。

证据基础：SCoT 在 HumanEval/MBPP 上 Pass@1 +13.79%；Devil's Advocate 在 WebArena 上无效重试 -45%；Few-shot 校准是 Anthropic 实测主力。所有手段与后续方向（思考预算、评估扇出、BoN、ThinkMerge）正交且互补。

① 角色化推理引导模板（始终注入）

按 X-Agent-Role 自动在 system prompt 尾部追加对应模板。

Planner — 发散式 + 完备性导向

5 步法：① 枚举所有隐含能力（对抗遗漏） → ② 每个能力天真实现会出什么问题（Pre-mortem） → ③ 区分 [MVP] / [LATER]（防过度工程） → ④ 识别依赖序（避免 Generator 执行顺序错误） → ⑤ 先推理再产出 JSON

Generator — SCoT 结构化思维链

5 步法：① INPUT/OUTPUT 明确（避免实现偏移） → ② CONTROL FLOW 用顺序/分支/循环结构描述（SCoT +13.79% Pass@1） → ③ HARD PARTS 识别并描述方案 → ④ EDGE CASES 至少 3 个 → ⑤ 实现代码应直接反映控制流

Evaluator — Devil's Advocate 魔鬼代言人

5 维度协议：① SPEC COMPLIANCE（逐条验收标准比对，标 PASS/FAIL+证据） → ② ATTACK SURFACE（每个函数找一个破坏性输入） → ③ ERROR HANDLING（每个外部调用是否有容错） → ④ IMPLICIT ASSUMPTIONS（环境假设是否成立） → ⑤ REGRESSION RISK。

起手句："Assume this code contains bugs." 输出分级：CRITICAL / MAJOR / MINOR

场景Proxy 行为理由

③ Budget Forcing 三级实现

Level A：System Prompt 级 Day 1

所有角色始终注入："After initial analysis, pause and reconsider: what did I miss? What assumption might be wrong?"

零工程成本

Level B：输出检测 + 自动追问 Day 4-5

检测信号：响应过短（代码实现请求 < 200 tokens）、无 <think> 块、直接给结论。触发后追加一轮："Your response was too brief. What design decisions did you make and why? What alternatives did you reject? Revise."

阈值按 intent 分别设定；最大追问 1 次

Level C：后处理质量门控 实验

静态分析 Generator 输出：TODO/FIXME 占位符（未完成）、空 catch 块（error swallowing）、函数数量与 spec 不匹配。检测到后注入针对性修复指令。

⑤ Pre-mortem 分析注入

学术依据：Devil's Advocate（arxiv 2405.16334）WebArena 上减少 45% 无效重试。

在 Planner 完成 feature 分解后注入："Imagine this plan has already failed. 3 most likely causes: ① Technical risk (hardest part generator will get wrong) ② Scope risk (seems simple but has hidden complexity) ③ Integration risk (individually correct but fails combined). Add mitigation notes."

简化版注入 Generator 复杂 feature 前："Quick pre-mortem: what ONE thing is most likely to go wrong? How will you prevent it?"

⑥ 上下文位置优化

解决 "Lost in the Middle" 效应：长上下文中模型对中间部分注意力最弱。

注入内容位置原因
验收标准 / spec 摘要system prompt 开头开头注意力强，适合北极星约束
角色定义 / 基础规则system prompt 中间常规内容
角色推理引导模板system prompt 末尾紧邻生成起始点，注意力最强
Few-shot 示例紧贴当前请求之前距任务最近

⑦ 负面约束注入（对抗「好学生综合症」）

模型倾向对长任务的危害对抗约束（肯定式改写）
过早宣布完成留下半成品"ALWAYS verify EVERY acceptance criterion before declaring complete"
过度工程消耗上下文"ONLY implement what was explicitly requested. No unrequested features or refactoring"
跳过错误处理Happy path 可用但遇错就崩"ALWAYS wrap external calls (API/DB/file) with error handling"
自评过于宽松Evaluator 变 Yes-man"Give PASS only with POSITIVE evidence for every criterion"

用肯定式（ALWAYS...）代替否定式（Don't...）——研究表明模型对肯定指令遵从率更高。放在 system prompt 末尾。

⑧ Few-shot 校准（Evaluator 专用）

注入 2-3 个标注示例，包含 FAIL 案例 + PASS 案例：

FAIL 示例：有 SQL 注入 + 无错误处理 + 无 null 检查 → CRITICAL+MAJOR → FAIL

PASS 示例：参数化查询 + try/catch + null 检查 → 仅 MINOR → PASS

必须包含 FAIL——只有 PASS 会训练出宽松判定。按项目域定制（Web vs CLI vs 数据管道）。2-3 个足够，超过 5 个边际递减。

⑨ 结构化输出强制

Planner → JSON Feature List

每个 feature 必含：id, name, priority(MVP/LATER), acceptance_criteria[], dependencies[], complexity, risk_note

Evaluator → JSON Report

包含：verdict(PASS/FAIL), issues[{severity, category, location, description, suggestion}], verified_criteria[]

机器可读 → Agent 框架直接消费 → 无需二次解析 → 多视角扇出聚合时可自动合并 issues

7.3 思考预算编排

Proxy 按 X-Agent-Role + X-Agent-Intent 自动设置 max_thinking_tokens：

角色 + 意图起步值说明
Evaluator（代码审查）8192审查需要系统性思考
Planner（任务分解）8192开放式推理
Generator（复杂实现）4096编码推理
Generator（简单编辑）1024短修改
Generator（工具调用）0读文件、搜索不需要思考
调参方法：在 Golden Set 上对同一任务跑不同预算，绘制质量-延迟曲线，找边际收益拐点。Qwen 的 <think> 机制与 Claude 不同，过长可能出现重复推理或自我矛盾——需设上限并监控。

关键区分：反馈必须是外部事实（编译错误、测试结果），不是 LLM 对 LLM 的意见。研究表明：无外部反馈的 Self-Refinement 效果很差甚至有害。

7.5 Best-of-N（可选 — 非全局默认）

并行向 Qwen 发送 N 个请求（不同 temperature/seed），利用 KV-Cache 共享降低开销。

选择方式适用场景可靠度
结构化验证器代码能编译？JSON 合法？测试通过？最高 — 无幻觉
自洽性投票有确定答案的推理题高 — 不需额外模型
序列 logprob通用中 — 高 logprob ≠ 高质量
何时开启 BoN：① Evaluator 连续 N 轮不通过 → 换思路重新生成；② Planner 分解质量影响全局 → 多采样；③ 安全/合规代码 → 高标准。不做全局默认。

7.6 轻量版并行思考（多视角提示 — 实验级）

受 ParaThinker 启发，但不需要训练模型，Proxy 层即可实现：

Proxy 收到复杂请求 │ ├─路径 A：system prompt + "Approach by identifying edge cases first" ├─路径 B：system prompt + "Approach by designing the data flow first" └─路径 C：system prompt + "Approach by writing tests first, then implementing" │ ├─ 等待所有路径完成 │ └─选择策略： ├─ 有验证器 → 选通过验证的 ├─ 无验证器 → 交给 LLM 综合（类似 ThinkMerge 总结阶段） └─ 或用 logit 融合（若已实现 ThinkMerge）
为什么比纯 BoN 更好：纯 BoN 同一提示不同温度 = 在同一解空间重复采样。多视角提示 = 引导模型探索不同解空间。成本相同，信息量更大。

7.7 ThinkMerge — Logit 平均融合（探索级）

核心机制："Think in Parallel, Answer as One"

Step 1：并行生成 K 条独立思考路径（<think> 阶段各自推理）

Step 2：进入作答阶段后，每一步解码时对 K 条路径的 Logit 做算术平均

Step 3：用融合概率分布采样下一个 token → 反馈注入所有路径，保持后续一致

为什么对代码生成特别有价值

维度Best-of-N投票ThinkMerge
融合层级结果级答案级Token 级
开放式任务需要验证器选 — 代码无唯一正确答案无法投票生成过程本身综合多条思路
实验结果依赖选择器质量仅限封闭题LiveCodeBench 显著提升 Pass@1

vLLM 工程可行性

✓ 通过自定义 LogitsProcessor 实现，不修改 vLLM 核心引擎

✓ 多路径作为同一 batch 的不同序列提交，共享 KV-Cache 前缀

✓ 兼容标准解码策略（Top-k, Top-p, temperature）

⚠ 需确保路径同步完成当前步骤；异步对齐是主要工程挑战

⚠ K 条路径 GPU 计算量 ≈ K 倍（KV-Cache 共享可部分降低）

建议：先在离线实验环境用 Qwen 122B 跑 Golden Set 验证效果，确认 Pass@1 提升幅度后再决定是否投入工程化。若提升 >5%，值得做。

7.8 前沿研究启发（跟踪但不直接投入）

ParaThinker（清华）

引入可训练的 <think i> 控制 token，模型在单次前向传播中并行生成多条路径。1.5B 模型平均准确率提升 12.3%。

启发：隧道视野（走错路后计算再多也难纠正）是长链路的核心问题。并行探索多条路径是根本解法。我们的「轻量版并行思考」即此思路的 Proxy 层近似。

Elastic Reasoning（Salesforce）

将推理拆为思考阶段 + 解决方案阶段，分别设独立 token 预算。思考被截断时模型仍能产出合理方案。

启发：避免「思考太多导致输出被截断」——Proxy 可以把 max_thinking_tokens 和 max_tokens 分开控制。这对 Qwen 高 thinking 预算时的实际风险是有效缓解。

Learned Continue-Thinking Token

通过 RL 仅训练一个 <|continue-thinking|> token 的嵌入（冻结模型参数），在数学推理上优于固定 "Wait" token。

启发：如果 Budget Forcing 的 token 级方案上线，可考虑用 RL 学一个最优 token 而非硬编码 "Wait"。低成本（仅训练一个嵌入）。

7.9 Proxy 层落地清单

第一批 B0（1-2 天）
① 角色推理模板注入：按 X-Agent-Role 自动注入对应引导 prompt
② 思考预算按角色分配：max_thinking_tokens 按 role + intent 设值
③ Evaluator 严格度校准：few-shot 示例 + 系统提示注入
第二批 B1（3-5 天）
④ 多视角评估扇出：不同维度模板并行
⑤ 反思注入（条件触发）：Generator 完成 feature 后自动追加
⑥ Budget Forcing 方式 B：检测浅层输出时自动追加一轮
第三批 — 实验级
⑦ 轻量版并行思考：同一请求用不同视角提示并行发出，综合结果
⑧ ThinkMerge PoC：vLLM LogitsProcessor 自定义实现，Golden Set 验证

7.10 算力消耗策略（整合视图）

请求进来 │ ├→ 简单任务 → 122B thinking OFF + 角色模板注入 │ (算力 ≈1.2x，速度：快) │ ├→ 中等任务 → 122B thinking ON + 角色引导 + 验证 │ (算力 ≈2-3x，速度：正常) │ ├→ 复杂任务 → 122B + 反思注入 + 多视角评估扇出 │ (算力 ≈3-5x，速度：正常，评估并行) │ └→ 极难/反复失败 → 轻量并行思考(3视角) + BoN + 交叉评估 (算力 ≈6-10x，启用 MiniMax 交叉验证) 空闲时段 ├→ Golden Set 回归跑批 ├→ ThinkMerge 效果实验 └→ 思考预算-质量曲线绘制

八、实施路径

8.1 实施原则

MVP 优先

先跑通「一 feature → 工具验证 → 评估反馈 → 修复」，再加重平台能力。

Build to Delete

为当前模型弱点写的补丁要可删除。持久的是契约、指标、评估集。

实测驱动

思考预算数值、扇出维度数、校准方案——全部用 Golden Set + 日志决定，不拍脑袋。

算力投评估

优先评估与规划相关调用，生成侧扩展采样仅作补充。

8.2 实施路线

路线 A：Agent 框架

A1 — 契约与产物
定义 feature_list、progress、commit 约定。新 session 先读产物再动手。
A2 — 三角色闭环
Planner → Generator（单 feature）→ Evaluator → 循环直至通过或达重试上限。
A3 — 工具与真实验证
测试、构建、Playwright 等；Evaluator 输出必须可定位、可操作。

路线 B：Proxy（可与 A2 并行）

B0 — 基础骨架（1-2 天）
解析 X-Agent-Role/Intent → 注入思考预算 → 评估请求优先级 → 严格度 prompt 注入（直通模式，暂不扇出）。
B1 — 多视角扇出 MVP（3-5 天）
评估请求扇出 3 个并行维度 → 聚合（最严原则 + 去重）→ 返回结构化报告。含降级策略。
B2 — 校准与调优（持续）
Golden Set 建设 → Few-shot 校准 → 指标看板 → 思考预算实测调参。

路线 C：独立服务（有第一版 Proxy 指标后启动）

C1 — 最小仪表盘
按角色 token/成本、评估通过率趋势、长任务完成率。
C2 — 回归管道
模型升级前后跑同一 Golden Set，记录 diff。6 月模型切换时首次启用。

8.3 Proxy 层 API 契约

HTTP Header必选说明
X-Agent-Role是planner / generator / evaluator
X-Agent-Intent否影响思考预算：implement-feature / evaluate-feature / read-only-tool 等
X-Eval-Perspectives否评估维度列表，默认 functionality,edge_cases,code_quality
X-Eval-Strictness否low / medium / high / maximum，默认 high
X-Task-ID / X-Feature-ID否分布式追踪与评估历史关联

未带头时，Proxy 行为与直连模型一致（向后兼容）。

九、风险与缓解

风险影响缓解
评估过严 → 循环不收敛 Generator 反复修改无法通过，上下文爆炸 最大重试次数；严重度分级（仅 CRITICAL 阻断）；繁忙时降级扇出维度
纯 LLM 评估假装有工具 LLM 猜测边缘行为 = 更花哨的幻觉 Agent 层必须先落地工具链；Proxy 评估增强建立在工具产出之上
同模型自偏好 Evaluator 对 Generator 产出系统性偏松 工具事实 > LLM 判断；few-shot 校准；交叉评估；Golden Set 监控
集群忙时评估扇出抢资源 延迟升高，影响正常任务 监控 P95 排队；超阈值自动降级（减维度 / 关扇出）
思考预算数值拍脑袋 过高浪费、过低无效、Qwen 机制不同于 Claude 起步用保守值；在 Golden Set 上实测质量-延迟曲线，找拐点
Build to Delete 失败 补丁累积、系统越来越复杂 每次模型升级做「保留/简化/删除」评审；文档标注哪些是临时补丁

里程碑自检

✓ Agent 最小可演示

结构化 feature 列表可读入；一次一 feature；Evaluator 独立于 Generator；可从产物恢复。

✓ Proxy B0 完成

20+ 条 Golden Set；模型升级时跑回归并记录 diff。

十一、总结

为什么做：长时自主任务是集群唯一的真正差异化；空余算力是战略资产，Harness 是把它转化为可靠产出的唯一路径。

重点做什么：Agent 侧三角色（Planner + Generator + Evaluator）+ 结构化产物 + 工具验证；Proxy 侧角色化思考预算 + 评估增强（多视角 × 校准 × 优先级）+ 提示词驱动推理增强（角色模板注入、Budget Forcing、反思注入）。算力优先投评估和推理引导，不是冗余生成。

算力转化路径：提示词推理增强（零成本立即做）→ 思考预算编排 → 迭代精炼（带外部反馈）→ 多视角评估扇出 → 轻量版并行思考（实验）→ ThinkMerge Logit 融合（探索）。

怎么做：MVP → 实测 → 校准 → 渐进扩展。Build to Delete。6 月模型升级时做全面评审。所有数值不拍脑袋，用 Golden Set 和日志说话。
